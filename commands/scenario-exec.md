# /scenario-exec

Execute Gherkin test cases. Uses a streaming incremental mode (similar to Node.js Stream) to synthesize MixedCase instruction packets per Scenario, launching `scenario-case-runner` sub-agents immediately upon synthesis.

> **Path convention**: Template files referenced by this command are located in the skill `scenario-test` installation directory (hereinafter referred to as `{SKILL_DIR:scenario-test}`). The actual path varies by host:
> - Cursor: `.cursor/skills/scenario-test/`
> - Claude Code: `.claude/skills/scenario-test/`
> - Other hosts: Refer to the host's skill installation directory

## Command Parameters

```
/scenario-exec [cases=<.feature file path or Scenario fragment>] [config=<config file path>]
```

- `cases`: Optional. Specify which cases to run:
  - File path (e.g., `tests/features/search/product.feature`)
  - Directory path (recursively scans all `.feature` files in the directory)
  - Scenario title or text fragment (AI matches the corresponding Scenario in the case library)
  - **Default**: Auto-scans all cases under `{testDir}/features/`
- `config`: Optional. Specify the path to `scenario-run-config.md` (**Default**: Auto-finds `{testDir}/features/scenario-run-config.md`)

> Other runtime parameters (concurrency, max failure tolerance, tags filter, etc.) are configured in `scenario-run-config.md`, not passed via command line.

---

## Core Constraint: Main Model Must Never Execute Test Steps Directly

> ⚠️ **This constraint applies throughout the entire command execution, with no exceptions.**

**The main model's responsibilities are limited to**:
1. **Analysis/Investigation**: Reading files, analyzing reports, inferring failure causes
2. **Synthesizing MixedCase**: Translating Gherkin steps into specific tool call instructions, generating instruction packets
3. **Launching Runner sub-agents**: Executing all tests and investigations through `scenario-case-runner`
4. **Aggregating reports**: Extracting results from sub-agents' text reports, writing to result.json / summary.md

**The main model must NOT**:
- Directly call playwright-cli, curl, bash, or any other tool to execute test steps
- Use "just run it directly" as an investigation method

**If investigation is needed** (e.g., analyzing failure causes, verifying a hypothesis):
- Synthesize a dedicated "investigation MixedCase" (can focus on key steps, more concise than formal MixedCase)
- Launch a `scenario-case-runner` sub-agent for execution, marking the MixedCase's `caseId` with the `inv-` prefix (e.g., `inv-case-7f3a2c`)
- After the sub-agent reports back, the main model analyzes the investigation results to improve MixedCase generation logic
- ⚠️ Investigation run results are **not counted** in the formal summary (not written to result.json / summary.md)
- ⚠️ Investigation run count is not constrained by `maxRetries`, but should be kept reasonable (typically 1-2 times)

## Phase 1: Detection and Configuration Check

### 1.1 Infer testDir

Check in priority order (**independent of config file**, pure detection):
1. Does a `tests/` directory exist in the project root?
2. Does an `e2e/` directory exist in the project root?
3. Does a `test/` directory exist in the project root?
4. If none of the above exist, fall back to `tests/`

Result: `SCENARIO_TEST_DIR` (e.g., `tests`). Config file path: `{SCENARIO_TEST_DIR}/features/scenario-run-config.md`.

### 1.2 Check Configuration File (Idempotent)

Check if `{SCENARIO_TEST_DIR}/features/scenario-run-config.md` exists:

**If it doesn't exist** (typically this file is auto-generated by the skill when first adding cases; this is fallback logic):
1. Copy from `{SKILL_DIR:scenario-test}/run-config-template.md` to `{SCENARIO_TEST_DIR}/features/scenario-run-config.md` (create `{SCENARIO_TEST_DIR}/features/` directory if it doesn't exist)
2. **Pause execution**, show the user:
   ```
   Generated {SCENARIO_TEST_DIR}/features/scenario-run-config.md. Please fill in the executor configuration and re-run:
   - command: Executor command (defaults to playwright-cli, modify as needed)
   - config: Describe your executor configuration in natural language (target address, authentication method, etc.)
   ```
3. Wait for user confirmation before re-running (do not continue execution)

**If it exists**: Read all configuration fields and continue execution. If the config file has a non-empty `testDir` field, use it to override the 0.1 detection result.

### 1.3 Read Configuration

Extract the following variables from `scenario-run-config.md` (subsequent steps use **concrete values**, not variable names):
- `SCENARIO_TEST_DIR`: Actual testDir path
- `SCENARIO_CONCURRENCY`: Concurrency count (default 4)
- `SCENARIO_MAX_FAILURES`: Maximum failure count (default 10)
- `SCENARIO_MAX_RETRIES`: Maximum smart retry count (default 1, 0 means disabled)

From the `executor` config block:
- `EXECUTOR_COMMAND`: Actual executor command value (e.g., `playwright-cli`, `bash-curl`, `vitest run --reporter=json`)
- `EXECUTOR_CONFIG`: Free-text content of the `executor.config` field (entire text, preserved as-is, referenced during instruction packet synthesis)

From the `## Runtime Notes` section:
- `RUNTIME_NOTES`: Free-text content of the `notes:` field (entire text, preserved as-is); if empty or field doesn't exist, treat as no notes
- When synthesizing each MixedCase, **the `RUNTIME_NOTES` content must be applied to the instruction packet** (e.g., adding steps, adjusting wait times, etc.)

> **Note**: Both `executor.config` and `notes` are free text with no forced parsing into fixed fields. The main model, when synthesizing instruction packets, understands and extracts needed information from `EXECUTOR_CONFIG` based on the `EXECUTOR_COMMAND` type (e.g., baseUrl, authentication method), while also injecting universal improvements from `RUNTIME_NOTES` into each step sequence.

Generate a unique identifier for this run:
- `RUN_ID`: Format `run-{YYYYMMDD}-{HHmmss}`, e.g., `run-20260304-143021`
- `RUN_ARTIFACT_BASE`: `{SCENARIO_TEST_DIR}/test-results/{RUN_ID}/` (full absolute path)

## Phase 2: Case Discovery

### 2.1 Scan Case Files

Determine scan scope based on parameters:
- If `cases=<file path>` is provided: Use that file directly
- If `cases=<directory path>` is provided: Recursively scan all `.feature` files in that directory
- If `cases=<Scenario title or text fragment>` is provided: Recursively search under `{SCENARIO_TEST_DIR}/features/`, matching Scenario titles or step text
- If no parameters: Scan all `.feature` files under `{SCENARIO_TEST_DIR}/features/`

**Scan result**: An ordered list of `.feature` files (`FEATURE_FILES`)

### 2.2 Tags Filter (If `tags` field is configured in run-config)

If `scenario-run-config.md` has a `tags` filter expression configured, filter each scanned Scenario with the following logic:
- `@P0`: Scenario contains the `@P0` tag
- `@P0 and @web`: Contains both tags (AND)
- `@P0 or @api`: Contains either tag (OR)
- `not @skip`: Does not contain this tag (NOT)
- Supports parentheses: `@P0 and (@web or @api)`

If the tags expression has a syntax error, stop immediately and report the syntax error; do not launch any Runners.

### 2.3 Display Execution Plan

Show the user a summary:
```
About to execute:
  Case count: N Scenarios (from M .feature files)
  Executor: {EXECUTOR_COMMAND}
  Concurrency: {SCENARIO_CONCURRENCY}
  Artifact directory: {RUN_ARTIFACT_BASE}
```

**Concurrency compatibility warnings** (only triggered when `SCENARIO_CONCURRENCY > 1`):

- **playwright-cli**: Each MixedCase's `[Environment Setup]` creates an independent browser process via `-s=<caseId>` named session, providing natural process-level isolation, safe for concurrency:
  ```
  ✅  playwright-cli concurrent execution: Each Runner operates in an independent session (-s=<caseId>), process-level isolation, no shared browser state.
      Setup creates the session, Teardown precisely closes it, no concurrency risk.
  ```
- **Custom CLI**: If `EXECUTOR_CONFIG` describes starting a local service (contains keywords `port`, `dev server`, `localhost`), the temporary directory isolation in `[Environment Setup]` may not be sufficient to resolve port conflicts; warn:
  ```
  ⚠️  CLI executor starts a local service; port conflicts may occur during concurrent execution.
      Consider setting concurrency to 1, or configure port isolation parameters in executor.config (e.g., --port=auto).
  ```
- **bash-curl**: `curl` is naturally stateless, no warning needed.

## Phase 3: Batch Parallel Execution

> **Core principle: Parallelize whenever possible** — all Runners launch simultaneously, never waiting on each other. `concurrency` serves only as an upper limit on simultaneous in-flight count, not a reason for serialization.

### 3.1 Pre-scan: Get Full Scenario List

Read each `.feature` file in `FEATURE_FILES`, **parse all Scenarios at once**, producing the complete execution list `SCENARIO_LIST`.

Scenario parsing rules: Identify `Scenario:` keyword blocks (to the next `Scenario:` or end of file), recording `featurePath` + `scenarioTitle` + `steps`.

### 3.2 Batch Synthesis + Parallel Launch

**Generate MixedCase instruction packets for all Scenarios in `SCENARIO_LIST` at once, then launch all Runners simultaneously**:

1. Synthesize the corresponding MixedCase instruction packet for each Scenario (filled according to the rules below)
2. **Launch all `scenario-case-runner` calls simultaneously** (parallel tool calls, don't wait for any to complete)
3. Wait for all Runners to return results, then aggregate

If `SCENARIO_LIST` count exceeds `SCENARIO_CONCURRENCY`: Launch the first `SCENARIO_CONCURRENCY` Runners (simultaneously), wait for that entire batch to complete, then launch the next batch (simultaneously), until all are processed. Each batch is still internally parallel.

#### Synthesizing MixedCase Instruction Packets

> **Important**: Must follow the format defined in `{SKILL_DIR:scenario-test}/mixed-case-template.md`, filling all fields. **All fields must contain concrete values; `{variable}` style placeholders are prohibited. Instruction packets must not contain any file paths.**

Fill instruction packet fields:

1. **Basic Information**:
   - `caseId`: Stable short hash based on `featurePath + scenarioTitle`, format `case-{6chars}`
   - `scenarioTitle`: Original Scenario title text

2. **Execution Environment** (all replaced with concrete values, extracted from `EXECUTOR_CONFIG` free text):
   - `executor`: Use the actual value of `EXECUTOR_COMMAND` (e.g., `playwright-cli`)
   - Based on `EXECUTOR_COMMAND` type, understand and extract needed environment info from `EXECUTOR_CONFIG` text:
     - **playwright-cli**: Extract target address (as `baseUrl`), authentication method (as `storageStatePath` or `authHeader`)
     - **bash-curl**: Extract API address (as `baseUrl`), Authorization header
     - **Custom CLI**: Extract CLI-required environment variables, config parameters, etc.
   - Write extracted information as concrete values into the instruction packet; **copying `EXECUTOR_CONFIG` verbatim into the packet is prohibited**

3. **Execution Steps** (each step is a concrete tool call instruction, "spoon-fed" level):

   For each Given/When/Then/And/But step in the Scenario:
   - **All steps uniformly use `EXECUTOR_COMMAND`** (single executor, from run-config `executor.command`, already a concrete value)
   - **Translate into concrete tool call instructions** (main model's responsibility):
     - Translate Gherkin natural language descriptions into specific tool calls (tool name + full parameters)
     - Replace all parameters with concrete values (e.g., keep `"headphones"` as-is, replace target address extracted from `EXECUTOR_CONFIG` with actual URL)
     - If a step involves waiting, specify the wait strategy (e.g., wait for element to appear, wait 2 seconds, etc.)
     - If a step involves scrolling, specify scroll direction and target
   - **Specify assertion target**: Expected observable result (specific to page elements or response fields)
   - **Do not write any file paths into the instruction packet** (screenshots, result files, etc. are handled by the main model after sub-agent reports)

   **Gherkin Step Translation Rules (T606)**:

   Based on Gherkin step semantics, translate into specific tool calls for the corresponding `EXECUTOR_COMMAND`:

   **When `EXECUTOR_COMMAND` is `playwright-cli`**:
   > `playwright-cli` is called via bash command line; all commands include the `-s=<caseId>` parameter to operate on the independent session created in Setup.
   > Before interaction, typically `snapshot` first to get element refs, then interact using refs.

   | Gherkin Semantics | Translated Tool Call |
   |---|---|
   | User visits/opens page URL | `bash: playwright-cli -s=<caseId> goto <target address from EXECUTOR_CONFIG>/path` |
   | Check current page state (before interaction) | `bash: playwright-cli -s=<caseId> snapshot` (get element list and refs) |
   | User clicks an element | First `snapshot` to confirm ref, then `bash: playwright-cli -s=<caseId> click <ref>` |
   | User fills in/types text | First `snapshot` to get input field ref, then `bash: playwright-cli -s=<caseId> fill <ref> "input content"` |
   | User presses a key (e.g., Enter) | `bash: playwright-cli -s=<caseId> press Enter` |
   | User waits for a condition | Retry `snapshot`, verify target content appears in output (max N times, 1s interval) |
   | Assert page contains certain text | `bash: playwright-cli -s=<caseId> snapshot`; Assertion target: snapshot output contains `"expected content"` |
   | Load SSO login state | `bash: playwright-cli -s=<caseId> state-load <storageStatePath>` (call after open) |

   **When `EXECUTOR_COMMAND` is `bash-curl`**:
   | Gherkin Semantics | Translated Tool Call |
   |---|---|
   | Send GET request | `bash: curl -s -X GET "<API address from EXECUTOR_CONFIG>/path" -H "<Authorization header>"` |
   | Send POST request (with body) | `bash: curl -s -X POST "<API address>/path" -H "Content-Type: application/json" -d '{"key":"value"}'` |
   | Assert response field | Tool call: above curl + `| jq '.field'`; Assertion target: output equals `"expected value"` |

   **When `EXECUTOR_COMMAND` is a custom CLI (e.g., `vitest run`)**:
   - For each Scenario, construct the complete CLI call command
   - Example: `bash: npx vitest run --testNamePattern="scenario title" --reporter=json`
   - Assertion target: CLI exit code is 0, or output contains `"passed"` keyword

#### Synthesizing [Environment Setup] and [Environment Teardown]

Each MixedCase instruction packet **must include** `[Environment Setup]` and `[Environment Teardown]` sections to create and clean up isolated test execution environments. The sub-agent guarantees Teardown always executes (like try/finally).

Fill based on `EXECUTOR_COMMAND` type:

Fill based on `EXECUTOR_COMMAND` type, **Teardown must precisely clean up resources created by Setup; fuzzy cleanup (e.g., `playwright-cli close-all`) is prohibited**:

**playwright-cli**:
- **Setup**: Inject `bash: playwright-cli -s=<caseId> open <baseUrl>`; `Captured Output: none` (session name = caseId, already known)
- **Teardown**: `Capture Dependency: none`; Inject `bash: playwright-cli -s=<caseId> close` (precisely close this Runner's session, not affecting other Runners)
- Even with `concurrency = 1`, Setup/Teardown must be injected to ensure clean browser process exit after each execution
- If config has `storageStatePath`, in Setup: first `open` then `bash: playwright-cli -s=<caseId> state-load <storageStatePath>`

**bash-curl**:
- **Setup / Teardown**: Both filled with `"none"`, `curl` is naturally stateless, no isolation needed

**Custom CLI** (e.g., `vitest run`, `jest --ci`):
- **Setup**: Inject `bash: mktemp -d /tmp/scenario-${caseId}-XXXXXX`; `Captured Output: SETUP_TMPDIR` ← mktemp's output path
- **Teardown**: `Capture Dependency: SETUP_TMPDIR`; Inject `bash: rm -rf "<SETUP_TMPDIR>"` (precisely delete that directory, not affecting other directories)
- If `EXECUTOR_CONFIG` has no local service startup (no `port`/`dev server` keywords), Setup/Teardown can be simplified to `"none"`

#### Parallel Launch of All Runners

After all MixedCase instruction packets are synthesized, **launch all `scenario-case-runner` calls simultaneously** (parallel tool calls, not serial waiting):

- If `SCENARIO_LIST` count ≤ `SCENARIO_CONCURRENCY`: Launch all at once, all parallel
- If `SCENARIO_LIST` count > `SCENARIO_CONCURRENCY`: Process in batches, each batch simultaneously launches `SCENARIO_CONCURRENCY` Runners, waits for the entire batch to complete before launching the next batch

> ⚠️ **Serialization is strictly prohibited**: Do not launch one Runner and wait for it to complete before launching the next. All Runners within the same batch must be simultaneously in-flight.

Each Scenario is assigned an independent sequence number `SEQ` (starting from 1, incrementing), used for progress output.

**Early Stop Check (T506, only applies to multi-batch scenarios)**:
- Maintain a global failure counter `FAIL_COUNT` (initially 0)
- Whenever a sub-agent reports failure (`failed` or `error`): increment `FAIL_COUNT`
- **Check timing**: After each batch completes, decide whether to launch the next batch
- If `FAIL_COUNT` ≥ `SCENARIO_MAX_FAILURES`:
  - Do not launch subsequent batches
  - Enter Phase 5, marking `earlyStop: true` in the summary
- Note: Runners within the same batch are already launched in parallel and cannot be cancelled mid-flight

### 3.3 Real-time Progress Feedback

Whenever a Scenario completes, output a progress line:
```
[✓ passed] case-7f3a2c  Search for existing product  (1.2s)
[✗ failed] case-3b9d1e  Login failure message         (assertion failed: expected error message but page unchanged)
```

## Phase 4: Smart Retry (Executed After Each Batch Completes)

> **Timing**: After each batch of Runners completes, before launching the next batch (or entering Phase 5), execute this step.

### 4.1 Analyze Failed Cases

For cases in this batch with status `failed` or `error`, **analyze the report content one by one**, classifying each failure as:

| Classification | Identification Markers | Handling |
|---------------|----------------------|----------|
| `app-bug` | App behavior doesn't match the Scenario's expected functional result (e.g., assertion says "button click should redirect to login page" but it didn't redirect); this is a real business bug | No retry, keep failed status |
| `mixcase-fixable` | The MixedCase steps themselves have issues (e.g., missing SSO authentication step, incorrect step order, insufficient wait time, state pollution from uncleaned data); unrelated to whether the app functions correctly | Retry (see 4.2) |
| `env-error` | Execution environment issues (session startup failure, network unreachable, playwright-cli command itself errors); not a MixedCase problem | No retry, keep error status |

> **Key judgment principle**: MixedCase issues mean "the test script was written wrong"; app-bug means "the system under test has a problem". If uncertain, lean toward `app-bug` (don't blindly retry).

### 4.2 Generate Fix Plans for `mixcase-fixable` Cases

For each `mixcase-fixable` case, the main model generates a fix plan and determines if the fix is universal:

**If the root cause cannot be determined from the report alone, do an Investigation Run first**:
1. Synthesize a dedicated investigation MixedCase (can include only key steps, `caseId` prefixed with `inv-`, e.g., `inv-case-7f3a2c`)
2. Launch a `scenario-case-runner` sub-agent for investigation
3. After the sub-agent reports back, the main model analyzes the results to identify the root cause
4. Investigation results are not written to result.json / summary.md (strictly prohibited from being counted in formal summary)
5. Based on investigation conclusions, proceed to the fix plan generation below

**Universal improvement** (improvements applicable to all/most Scenarios):
- Example: All tests need to call `state-load` to load SSO login state before starting
- Example: Need to wait an extra 2s after navigation before interacting
- Example: A certain API requires specific request headers

→ **Action**:
1. Append an improvement entry to the `notes:` field in `scenario-run-config.md` (in natural language)
2. Regenerate this Case's MixedCase (applying the improvement)
3. Re-launch the Runner; mark in report as `retried: true, retryReason: "Universal improvement: ..."`

**One-off fix** (improvements only applicable to this Case):
- Example: Clean up leftover product data from last execution (only this Case is affected)
- Example: An element in this Case needs to be scrolled into view before clicking

→ **Action**:
1. **Do not write** to `scenario-run-config.md`
2. Directly reflect the fix in the regenerated MixedCase steps
3. Re-launch the Runner; mark in report as `retried: true, retryReason: "One-off fix: ..."`

### 4.3 Retry Constraints

- **Each Case can be retried at most `SCENARIO_MAX_RETRIES` times** (default 1): Cases that have reached the retry limit and still fail are recorded as their final status (`failed` or `error`), no further retries
- **When `SCENARIO_MAX_RETRIES = 0`**: Skip the entire Phase 4, do not perform any smart retry analysis
- **Retries count toward `maxFailures`**: Cases that still fail after retry are counted in the failure count
- **Retry Runners are supplementary to the current batch**: Retries and the next normal batch do not interfere with each other; retry Runners complete before entering the next batch

## Phase 5: Result Aggregation

### 5.1 Wait for All In-flight Tasks to Complete

Wait for all launched `scenario-case-runner` sub-agents to return text results.

### 5.2 Aggregate Case Results (T507)

Extract the following information from each sub-agent's **text report** (do not read any files):
- `caseId`: Case identifier
- `Status`: `passed` / `failed` / `error`
- `Steps executed`: Total step count
- `Duration`: ~X seconds
- `Step results`: Each step's passed/failed
- `Failure reason` (if failed): Failed step and error description

Statistical summary:
- `total`: Total Scenario count
- `passed`: Pass count
- `failed`: Failure count (including cases marked as `skipped` after early stop)
- `skipped`: Scenario count not executed due to early stop
- `error`: Sub-agent crash/exception count
- `durationMs`: Total duration (from first Scenario start to last completion)
- `earlyStop`: Whether maxFailures early stop was triggered

### 5.3 Write Summary Files

**result.json** (written to `{RUN_ARTIFACT_BASE}/result.json`):
```json
{
  "runId": "<RUN_ID>",
  "startTime": "<ISO 8601>",
  "endTime": "<ISO 8601>",
  "durationMs": <total duration>,
  "executor": "<EXECUTOR_COMMAND>",
  "concurrency": <SCENARIO_CONCURRENCY>,
  "summary": {
    "total": <N>,
    "passed": <N>,
    "failed": <N>,
    "skipped": <N>,
    "error": <N>
  },
  "earlyStop": <true|false>,
  "cases": [
    {
      "caseId": "<case-id>",
      "scenarioTitle": "<title>",
      "status": "passed|failed|error|skipped",
      "durationMs": <duration>,
      "failedStep": <failed step number or null>,
      "errorType": "<assertion|timeout|environment|tool-error|null>",
      "errorMessage": "<error description or null>"

}
```

**summary.md** (written to `{RUN_ARTIFACT_BASE}/summary.md`):
```markdown

**Environment**: {EXECUTOR_COMMAND}
**Concurrency**: {SCENARIO_CONCURRENCY}
```

```json

```

### 5.4 Output Exit Code

- `0`: All Scenarios passed
- `1`: Failures exist (failed > 0)
- `2`: Environment unavailable (baseUrl unreachable) or early stop (maxFailures triggered)
- `3`: System error (sub-agent crash, file write failure, etc.)

---

## Error Handling

- **Target address unreachable**: Perform a lightweight health check as the first step (GET target address, timeout 5 seconds); if unreachable, stop immediately and return exit code 2
- **Sub-agent crash**: Mark the case as `error`, does not affect other Runners, continue execution
- **Case syntax error**: Skip that `.feature` file, log a warning, continue processing remaining files
- **Artifact directory creation failure**: Stop immediately, return exit code 3, prompt user to check filesystem permissions

## Appendix: Extensible Executor Synthesis Guide (T904)

When `EXECUTOR_COMMAND` is a non-built-in executor (not `playwright-cli` / `bash-curl`), the main model's instruction packet translation rules:

### Custom CLI Executors

If `EXECUTOR_COMMAND` looks like `vitest run`, `jest --ci`, `playwright test`, or other CLI tools:

- Translation strategy: **Map each Scenario as a whole to a single CLI call** (not translating step by step)
- Tool call format: `bash: {EXECUTOR_COMMAND} --testNamePattern="<scenarioTitle>" [other params]`
- Assertion target: CLI exit code is 0 (or determined by CLI output)
- Wait/Interaction: Wait for CLI process to complete (timeout per `EXECUTOR_CONFIG` description or default 60 seconds)

### MidScene Executor

If `EXECUTOR_COMMAND` contains the `midscene` keyword:

- Translation strategy: Translate Gherkin steps into MidScene's natural language instructions (`.ai()` method calls)
- Step format examples:
  - `midscene: ai("Type 'headphones' in the search box and click the search button")`
  - `midscene: aiAssert("Search results list has appeared")`
- Assertion target: `aiAssert` does not throw an exception

### Extension Principles

- **Main model bears translation responsibility**: Regardless of executor type, the main model must translate Gherkin natural language into directly callable instructions for that executor
- **Sub-agent executes as instructed**: The sub-agent only needs to execute per the `Tool Call` field, no need to understand executor type
- **Declared in `run-config`**: Users declare the executor in the `executor.command` field; the main model selects translation strategy accordingly

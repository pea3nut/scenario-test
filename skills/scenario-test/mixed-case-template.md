# MixedCase Instruction Packet Format Template

> **This file is for `/scenario-exec` main model reference only. It is not exposed to users; users do not need to fill in or modify it.**
>
> When the main model synthesizes a MixedCase instruction packet, it must follow this format to fill all fields, ensuring the `scenario-case-runner` sub-agent can directly execute tool calls without any inference.
>
> **Key requirements**:
> - All fields must contain concrete values; **`{variable}` placeholders are prohibited**
> - **Instruction packets must not contain any file paths** (no artifact paths, result paths, etc.); the sub-agent writes no files
> - Each step's tool call instruction must be "spoon-fed" — including tool name, parameters, wait logic, assertion target — **not** a direct translation of the Gherkin original text
> - After receiving this packet, the sub-agent directly calls tools and reports each step's result in **plain text**

---

## Instruction Packet Format

```
=== MixedCase Instruction Packet ===

[Basic Information]
caseId: <stable unique identifier generated by the main model, based on hash of featurePath + scenarioTitle, e.g., case-a3f2b1>
scenarioTitle: <original Scenario title text, e.g., "Search for existing product">

[Execution Environment]
executor: <concrete executor command value, e.g., playwright-cli | bash-curl | vitest run --reporter=json>
<The main model understands and extracts environment info from run-config's executor.config free text, converting to concrete values here, for example:>
baseUrl: <target address extracted from executor.config, e.g., https://platform-boe.example.com (omit if executor doesn't need it)>
authHeader: <authentication header extracted from executor.config, e.g., "Bearer eyJ..." | "none" (omit if executor doesn't need it)>
<Other executor-specific parameters, added as needed; main model judges what information is useful for sub-agent execution>

[Environment Setup]
Description: Before executing test steps, the sub-agent must complete all operations in this section to create an execution environment fully isolated from other concurrent Runners.
      If any Setup step fails, stop immediately (do not enter [Execution Steps]), but [Environment Teardown] must still be executed.
      ⚠️ When using playwright-cli, the session name directly uses caseId (declared in [Basic Information]); no capture needed, Teardown can reference it directly.

setup-1:
  Tool Call: <concrete tool call to create isolated environment, e.g.:
    - playwright-cli: bash: playwright-cli -s=<caseId> open (creates an independent session for this Runner, process-level isolation, no shared browser state with other Runners)
    - bash: mktemp -d /tmp/scenario-XXXXXX (creates an isolated temp directory for CLI tools, command output is the directory path)
    - Fill "none" if no isolation needed>
  Wait/Interaction: <wait for environment ready, e.g.: Wait for playwright-cli session startup to complete (exit code 0) | none>
  Assertion Target: <observable sign that environment is ready, e.g.: playwright-cli command exit code is 0, session is ready | temp directory path is non-empty | none>
  Captured Output: <resource identifier extracted from this step's tool call return value, stored as a named variable for Teardown reference, e.g.:
    - playwright-cli: none (session name = caseId, already known, no capture needed)
    - SETUP_TMPDIR: mktemp's output directory path (used for Teardown to precisely delete that directory)
    - Fill "none" if no capture needed>

<If multiple Setup steps are needed, continue with setup-2, setup-3>

[Execution Steps]
The following steps have expanded Gherkin into directly executable tool call sequences.
Each step includes: tool call instruction (concrete tool name + parameters) + wait/retry logic + assertion target.
The sub-agent executes steps sequentially by number; when an assertion fails, it stops immediately and reports the failure reason.

Step 1:
  Tool Call: <concrete tool call, e.g.:
    - bash: playwright-cli -s=<caseId> goto https://platform-boe.example.com (playwright-cli navigates to target page)
    - bash: playwright-cli -s=<caseId> snapshot (get page snapshot, learn current elements and refs)
    - bash: playwright-cli -s=<caseId> click <ref> (precisely click element using ref obtained from snapshot)
    - bash: curl -X POST https://api.example.com/login -H "Content-Type: application/json" -d '{"user":"alice"}'
    - bash: vitest run tests/login.spec.ts --reporter=json>
  Wait/Interaction: <if needed, e.g.: target element appears in snapshot output (retry up to 3 times) | none>
  Assertion Target: <expected observable result, e.g.: page URL becomes /dashboard, navigation bar visible in snapshot | response status code is 200, body.token field is non-empty>

Step 2:
  Tool Call: <concrete tool call>
  Wait/Interaction: <wait/scroll/retry instructions | none>
  Assertion Target: <expected result>

Step N:
  Tool Call: <concrete tool call>
  Wait/Interaction: <wait/scroll/retry instructions | none>
  Assertion Target: <expected result>

[Environment Teardown]
Description: The sub-agent must guarantee this section executes under all circumstances, equivalent to the finally block in a mandatory try/finally.
      ⚠️ The sub-agent must NOT exit before all Teardown steps complete; any error is a pause, not a termination.

[Teardown Execution Case Table — Sub-agent must cover all 4 cases]

| Case | Previous Phase Result | Teardown Behavior | Final Report Status |
|------|----------------------|-------------------|---------------------|
| ① Normal pass | Setup all succeeded → Execution steps all passed | Execute all Teardown steps | passed |
| ② Execution step failed | Setup all succeeded → Some execution step failed (subsequent steps skipped) | Execute all Teardown steps (do not skip) | failed |
| ③ Setup failed | Some Setup step failed (subsequent Setup steps skipped, execution steps not executed) | Only execute Teardown for **successfully completed Setup steps**; resources from unexecuted Setup steps don't need cleanup | error (setup-failed) |
| ④ Teardown itself failed | Any situation | Record the step as teardown-warning, **continue executing remaining Teardown steps** (do not interrupt) | Use status from above, do not downgrade |

      ⚠️ playwright-cli: Directly use `playwright-cli -s=<caseId> close` to precisely close this Runner's session; no capture variables needed.

teardown-1:
  Capture Dependency: <variable name captured from Setup step, e.g.:
    - playwright-cli: none (session name = caseId, reference directly, no extra capture variable needed)
    - SETUP_TMPDIR (bash: rm -rf needs this path to precisely delete the directory created by Setup)
    - Fill "none" if no dependency>
  Tool Call: <tool call to precisely clean up resources created by Setup, e.g.:
    - bash: playwright-cli -s=<caseId> close (precisely close this Runner's session, not affecting other Runners' sessions)
    - bash: rm -rf "<SETUP_TMPDIR value>" (precisely delete the temp directory created by Setup)
    - Fill "none" if no cleanup needed
    ⚠️ Writing playwright-cli close-all or other fuzzy commands that affect other Runners is prohibited>
  Wait/Interaction: <wait for cleanup to complete | none>
  Assertion Target: <observable sign that cleanup is complete (optional, failure only logs a warning)>

<If multiple Teardown steps are needed, continue with teardown-2, teardown-3>
=== End of Instruction Packet ===
```

## Step Expansion Principles (Main Model Must Follow)

Gherkin steps are typically high-level abstractions; the main model needs to consider these details when expanding:

### Concurrency Isolation Principle (Implemented via Setup/Teardown)

Each MixedCase must declare isolation boundaries through `[Environment Setup]` and `[Environment Teardown]`, ensuring concurrent Runners don't interfere with each other:

| Executor | Setup Injection | Captured Output | Teardown Injection |
|---|---|---|---|
| `playwright-cli` | `playwright-cli -s=<caseId> open [url]` | **none** (session name = caseId, already known) | `playwright-cli -s=<caseId> close` (precisely close this Runner's session) |
| `bash-curl` | none (`curl` is naturally stateless) | none | none |
| Custom CLI (e.g., `vitest`) | `mktemp -d /tmp/scenario-${caseId}-XXXXXX` | `SETUP_TMPDIR` ← mktemp output path | `rm -rf "<SETUP_TMPDIR>"` (precisely delete, not affecting other directories) |

> **Key constraint**: Teardown must NOT contain `playwright-cli close-all`, "clean up /tmp", or other fuzzy commands that affect other Runners. `playwright-cli` Setup/Teardown must be injected even when `concurrency = 1`.

### playwright-cli Step Expansion Notes

`playwright-cli` is called via bash commands; all commands include the `-s=<caseId>` parameter to operate on the current Runner's independent session:

| Gherkin Action Scenario | Expanded Must Include |
|---|---|
| Navigate to a page | `playwright-cli -s=<caseId> goto <url>` |
| Check current page state (required before interaction) | `playwright-cli -s=<caseId> snapshot` (get element list and refs) |
| Click a button/link | First `snapshot` to get ref → `playwright-cli -s=<caseId> click <ref>` |
| Type text | First `snapshot` to get input field ref → `playwright-cli -s=<caseId> fill <ref> "<text>"` |
| Wait for an element to appear | Retry `snapshot`, check if target ref appears in output (max N times, 1s interval) |
| Verify page text/element | `snapshot` and assert output text contains expected content |
| Load SSO login state | `playwright-cli -s=<caseId> open --config=<storage-state-path>.json` or `state-load` |

### bash Step Expansion Notes

| Gherkin Action Scenario | Expanded Must Include |
|---|---|
| Call an API endpoint | Complete curl command (including -X method, -H headers with auth, -d body) |
| Verify response fields | Parse response JSON with `jq` + assert field values |
| Run test suite | Complete CLI command + expected exit code |

## Filled Example

```
=== MixedCase Instruction Packet ===

[Basic Information]
caseId: case-7f3a2c
scenarioTitle: Search for existing product

[Execution Environment]
executor: playwright-cli
baseUrl: https://platform-boe.example.com
storageStatePath: .cursor/skills/sso-auth/playwright-storage-state.json

[Environment Setup]
Description: Start an independent session (-s=case-7f3a2c), ensuring this Runner is fully process-isolated from other concurrent Runners, each with its own browser process, cookies, and storage.

setup-1:
  Tool Call: bash: playwright-cli -s=case-7f3a2c open https://platform-boe.example.com
  Wait/Interaction: Wait for playwright-cli command to exit (exit code 0)
  Assertion Target: Command exits successfully, session case-7f3a2c is created and has opened the target page
  Captured Output: none (session name = caseId "case-7f3a2c", already known, no capture needed)

[Execution Steps]

Step 1:
  Tool Call: bash: playwright-cli -s=case-7f3a2c snapshot (get current page snapshot, confirm navigation succeeded and obtain element refs)
  Wait/Interaction: If navigation bar doesn't appear in snapshot, wait 1s and retry snapshot once
  Assertion Target: Snapshot output contains platform navigation bar text or page title, URL is platform-boe.example.com

Step 2:
  Tool Call: bash: playwright-cli -s=case-7f3a2c snapshot (confirm search box ref, e.g., e23) → bash: playwright-cli -s=case-7f3a2c fill e23 "headphones" → bash: playwright-cli -s=case-7f3a2c press Enter
  Wait/Interaction: After pressing Enter, wait ~2s, then execute snapshot to confirm search results loaded
  Assertion Target: Snapshot output contains product name text related to "headphones"

Step 3:
  Tool Call: bash: playwright-cli -s=case-7f3a2c snapshot (get search results list snapshot)
  Wait/Interaction: If results list is empty, wait 1s and retry snapshot once
  Assertion Target: Snapshot output contains product entries, count >= 1

[Environment Teardown]
Description: Regardless of whether the test passed, close session case-7f3a2c, release browser process resources, and avoid process accumulation affecting subsequent runs.

teardown-1:
  Capture Dependency: none (session name = caseId "case-7f3a2c", reference directly)
  Tool Call: bash: playwright-cli -s=case-7f3a2c close
  Wait/Interaction: none
  Assertion Target: playwright-cli command exit code is 0, session is closed (failure only logs a warning)
=== End of Instruction Packet ===
```
